Transparency notice

Intentional violations

The Reverse Accessibility Lab deliberately breaks WCAG 2.2 success criteria so visitors can feel inaccessible design firsthand. Every violation is scoped to one room and reversed once the lesson lands. This page documents each one.

For auditors and automated tools: violations on these pages are intentional and documented here. They are not regressions. The index page (/) and this page (/about/) are fully conformant.

Room 1 — Keyboard only

⌨️ Keyboard navigation
  • 2.1.1
    Keyboard — pointer events disabled on all interactive elements

    Nav links have pointer-events: none and the unlock button intercepts pointer and touch events via JavaScript, making every control mouse-unreachable. The lesson: everything works by keyboard — tab focus is intentionally enhanced with a strong visible glow.

  • 2.4.7
    Focus Visible — no hover styles, mouse cursor signals failure

    Hover states are deliberately absent and the cursor is set to not-allowed. Focus-visible styles are amplified instead, inverting the usual pattern of hover-rich, focus-minimal interfaces.

Room 2 — Screen reader

🔊 Semantic structure
  • 1.3.1
    Info and Relationships — heading hierarchy inverted

    The <h1> is visually tiny and muted while a <p> is styled to look like the main heading (large, bold, high-contrast). Screen readers announce the correct DOM order; sighted users read the visual lie.

  • 1.3.2
    Meaningful Sequence — CSS order scrambles visual layout

    Archive cards are assigned style="order: N" values that rearrange their visual position without touching the DOM. A screen reader reads the logical sequence; mouse users see a shuffled grid with no obvious pattern.

  • 1.4.3
    Contrast (Minimum) — section headings rendered below threshold

    Card <h3> elements are set to 0.7 rem in a dark muted colour (#5a5870) against the dark surface, falling well below 4.5:1. They are semantically significant but visually dismissible — the reverse of decorative-but-high-contrast elements that carry aria-hidden="true".

  • 2.4.6
    Headings and Labels — duplicate visible button labels

    Every access button reads "Access →" visually. Each has a distinct aria-label that identifies its target. Voice and screen-reader users can distinguish them; sighted users cannot.

Room 3 — Voice control

🎙️ Name, role, focus order
  • 2.4.3
    Focus Order — tabindex values create a non-sequential tab sequence

    Twelve controls are assigned explicit tabindex values that jump the keyboard cursor across sections in an illogical order, making keyboard navigation disorienting. One control — the correct target — has a unique, descriptive label.

  • 2.5.3
    Label in Name — buttons carry vague or duplicated visible labels

    Labels include "Click here" (×3), "Submit" (×2), "More info" (×2), "Go", "Option", and "Next" (×2) — none actionable for voice control. The target button's accessible name matches a specific, unique spoken phrase.

  • 1.3.1
    Info and Relationships — mouse-blocking overlay hides controls from pointer

    An aria-hidden="true" overlay sits above each interactive button at a higher z-index, intercepting mouse events. The accessibility tree exposes the real button beneath; pointer users hit a dead layer.

Room 4 — Zoom & magnification

🔎 Reflow & contrast
  • 1.4.10
    Reflow — critical control hidden in a narrow third column

    At 100% zoom a three-column grid keeps the "Acknowledge" button in a cramped annotations column with overflow: hidden, clipping it from view. At 200% zoom the viewport narrows below the grid breakpoint, columns stack, and the button becomes fully visible and operable — reflow reveals what bad layout hid.

  • 1.4.3
    Contrast (Minimum) — button and annotation text below threshold at normal zoom

    Annotation text is set to 0.67 rem in a near-background colour (#4a4860), and the button uses transparent background with a faint border — styled to blend with surrounding metadata so it reads as boilerplate rather than an action.

Room 5 — Reduced motion

🌀 Motion & vestibular
  • 2.3.3
    Animation from Interactions — continuous motion with no page-level off switch

    Five keyframe animations run on page load: bounce, spin, drift, pulse-border, and flicker — including on the button container itself. There is no on-page toggle; motion stops only when the OS prefers-reduced-motion: reduce setting is enabled. The lesson: the OS setting exists precisely for this, and enabling it unlocks the room.

  • 1.4.3
    Contrast (Minimum) — flicker animation drops panel values below threshold

    The flicker animation cycles panel value opacity from 1 down to 0.2, repeatedly dropping text contrast below 4.5:1 during the animation cycle.
